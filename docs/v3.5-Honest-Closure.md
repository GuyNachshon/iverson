# v3.5 Honest Closure

**Date:** 2026-04-28
**Status:** v3.5 is a working prototype that beats v2.5 on bt33 (toy game) but **does not generalize to real public ARC-AGI-3 games**. 0/9 public games tested showed any level completion. Stopping after 9 games because the pattern is unambiguous.

## What we built (recap)

- `agents/state_graph.py` — per-game state hashing + novelty signal.
- `agents/rules.py` — symbolic rule store with 4 effect families (global_delta, color_removed, level_progress, click_color_progresses, no_op). Per-effect support thresholds. Click-color-conditioned rules for ACTION6.
- `agents/saliency.py` — score objects by color rarity + edge proximity + smallness + aspect distinctness. Pick click target.
- `agents/iverson_v35.py` — wires it all: per-step picks `argmax(rule_score + saliency + novelty + jitter)`, with cycling-saliency click-target selection.
- `agents/skill_bank.py` — skeleton, not wired.

## What this tells us

1. **The bt33 win was specific to bt33's design** (2 button objects, equal saliency, click cycle reaches the right one). Public games have 5-30+ objects and the saliency cycle doesn't converge on the correct interaction target within 200 actions.

2. **All four approaches we've tried score 0 on public games**: random baseline, v2.5 (WM+entropy), v3 (terminal predictor), v3.5 (saliency+rules). The bottleneck is not architectural variation among these four — it's something they all lack.

3. **The missing capability is *persistence with hypothesis testing***. v3.5 re-decides every step, so even when it stumbles on the right action, the next step's saliency cycle picks a different target. Bt33's "click 4× to win" worked because eventually it cycled back to the right target enough times. Public games have non-trivial action sequences that need *committed execution*.

4. **The 200-action budget is real but not the primary issue.** Even on r11l where the per-level budget is 22 human actions and we have 200, we don't get 1 level. The agent isn't running out of time — it's failing to make progress.

## Honest verdict on the pivot

**The pivot to symbolic-first online learning was directionally correct** (per the literature on ARC-AGI-3-style benchmarks). The implementation, however, missed a key piece: the agent has no *plan execution* layer. It scores actions per step but doesn't commit to multi-step sequences.

**v3.5 is not Phase 2 baseline.** We don't have a Phase 2 baseline yet. All four agents we've built (random, v2.5, v3, v3.5) score 0 on real public games. We've been generating different ways to score 0.

## What I think comes next

Three concrete options, ranked by how likely I think they are to produce a non-zero score on a public game:

### Option 1: Add explicit hypothesize-and-test loops (CEL Rule Induction proper)

The CEL paper's rule-induction loop is the bit we sketched but didn't implement faithfully. Specifically:
- After every K actions, look at the trajectory and *generate hypotheses about what works* (text or symbolic).
- The next K actions *test* those hypotheses by committing to specific action sequences.
- If a hypothesis is confirmed, lock it in as a high-priority rule.

This is what gives the agent persistence. Right now the agent re-rolls saliency every step; CEL re-rolls hypotheses every K steps. **~3-4 days to build.** Highest probability of unlocking a non-zero score.

### Option 2: Domain-prior heuristics via VLM-extracted hypotheses

At env start, run a small VLM (Qwen2-VL-2B or Moondream2) on the initial frame to produce a structured guess: "This looks like a Sokoban variant; the player is the red square; the goal is the green square." Use that as the seed for rule induction.

This is the "VLM-as-prior" idea from the original Kaggle thread we discussed early on. It addresses *exactly* the "agent doesn't know what kind of game this is" failure mode. **~1 week to build (VLM bundling, prompt engineering, integration).** Medium probability — VLMs are mediocre at grid-game perception.

### Option 3: Step back and read the preview-comp winners' actual code

The Kaggle 2025 preview competition winners (StochasticGoose 12.58%, Blind Squirrel 6.71%) presumably published code. We've been reasoning about their approaches from descriptions. Reading their actual implementations might reveal tactical tricks we missed. **~1 day of research.** Low probability of unlocking, but high information value.

## Recommendation

**Do option 3 first** (1 day, low cost). If their code reveals a tactical insight we missed, apply it. Otherwise, do option 1 (CEL Rule Induction proper, 3-4 days).

I do not recommend pushing more code on the current v3.5 architecture without a fundamental change. The "rule store + saliency cycle + per-step argmax" loop has been demonstrated to not work on public games.

## What gets carried forward

- `agents/state_graph.py`, `agents/rules.py`, `agents/saliency.py`, `agents/skill_bank.py` — all retained. The pieces are sound; the *composition* needs work.
- Object-list representation (Phase 0b) — load-bearing, stays.
- The 4,700-trajectory corpus — useful for any future offline component.
- Phase 1 predictor checkpoint — demoted, optional dream-phase tool.

## What gets cut

- Continuing v3.5 evaluation. The 9-game result is enough.
- Skill bank wiring. Premature without a fix to the persistence problem.
- Any episode-boundary heuristic improvements without a hypothesize-and-test layer.

## Reset of expectations

We've now built four agent variants and all score 0 on public games. The Kaggle deadline is Nov 2 (6 months out). We have the foundation (object-list representation, env runner, training pipeline, evaluation infrastructure) but no working agent.

The honest path: read the preview-comp winners, then build option 1 (CEL-proper) carefully. Don't iterate on v3.5; the bottleneck is structural, not parametric.
