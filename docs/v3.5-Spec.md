# v3.5 Architecture Spec

**Status:** Spec, pre-implementation. Reflects the pivot from offline-pretrained terminal-state predictor to symbolic-first online learner per the Phase-1-Honest-Closure findings + research_v3 brief on CEL/WALL-E.

## Goal

Beat v2.5 (which scored 0/30 on five public ARC-AGI-3 games) by some non-zero margin within 200-action budget per game, on a 165ms/action wall-clock budget, no internet access at evaluation.

## Architecture name and tagline

**v3.5: state-graph search with online rule induction.**

The agent learns the current game's mechanics from its own first ~12 actions (rule store + state graph), then exploits that knowledge for the rest of the game. Rules persist across levels within a game (Competition Mode allows level resets, not game resets).

## What this is and isn't

**Is:** A symbolic-first online learner. Per-step decisions are heuristic + rule-store + state-graph lookup. Per-transition we extract one rule. Per-episode we promote successful action subsequences to a skill bank. No learned neural model on the critical path.

**Isn't:** A CEL clone (CEL's per-step LWM forward is 5s; we have 165ms). Not a WALL-E clone (WALL-E assumes a frontier-LLM prior; we have none). Not a v2.5 redo (v2.5 used pixel-CNN dynamics with no object-centric substrate; v3.5 uses object-list as the state representation throughout).

**Optionally:** a tiny online-trained novelty predictor (1-10M params, object-list → next-object-list) feeding the action scorer with "how surprised am I by this transition?" as an exploration bonus. **Not in the v3.5 MVP**; added in a v3.7 if MVP plateaus.

## Per-step decision flow (must fit ≤165ms wall-clock)

```
1. observe (FrameData → object-list Frame via existing converter)        ~1ms
2. state_hash = hash(object_list_signature)                                ~1ms
3. node = state_graph.get_or_create(state_hash, frame)                     ~1ms
4. for action in latest_frame.available_actions:
       score(action) = base
                     + rule_store.suggest(state, action) * α              ~5ms total
                     + saliency(state, action) * β                         ~3ms total
                     + novelty_bonus(state, action) * γ                    ~3ms total
                     + reset_penalty if action == RESET                    ~0ms
5. if all scores ~equal → uniform random over avail (epsilon fallback)
6. action = argmax(scores)
7. (If action is ACTION6) target_xy = saliency.click_target(state)         ~2ms
8. record_pending_transition(state, action)
9. return action
```

Total: well under 20ms in expectation. Comfortable.

## Per-transition update (after env returns next frame)

```
1. delta = diff(state_before, state_after)
2. rule = induce_rule(state_before, action, delta)
3. rule_store.add_or_update(rule, observed=True)
4. state_graph.add_edge(state_before_hash, action, state_after_hash)
5. if delta is non-trivial: clear novelty cache around state_before
6. (optional v3.7) novelty_predictor.update_buffer(state_before, action, state_after)
```

## Per-episode update (level WIN or GAME_OVER)

```
1. trajectory = list of (state, action, reward, next_state) for this level
2. extract_subsequences(trajectory)
   → for each contiguous slice that increased levels_completed, add to skill_bank
3. update_rule_confidences(rule_store, trajectory)
   → rules that fired and were correct: +confidence
   → rules that fired and were wrong: −confidence
4. boredom_state.reset()
5. (optional, episode-rate budget allows) call_local_LLM_to_summarize(rule_store)
   → Qwen 0.5B-4B at level boundaries; deferred to v3.7
```

## Data structures

### `agents/state_graph.py`

```python
@dataclass
class StateNode:
    state_hash: int                           # hash(object_list signature)
    visit_count: int = 0
    actions_tried: dict[int, ActionStats]     # action_id → stats
    salient_object_idxs: list[int] = []       # cached saliency picks
    
@dataclass
class ActionStats:
    times_tried: int = 0
    delta_sum: float = 0.0          # sum of |frame change| observed
    leads_to_win: int = 0            # how often this action led to a level win
    leads_to_game_over: int = 0      # how often it triggered GAME_OVER
    last_used_step: int = -1
```

State hash strategy: signature is `tuple(sorted((obj.color_id, round(obj.cx,2), round(obj.cy,2)) for obj in frame.objects[:32]))`. Truncating to 32 objects + rounding centroids to 2 decimals collapses near-identical states.

**Persistence:** state graph resets per game (between games we know nothing). Rule store and skill bank persist within a game across levels.

### `agents/rules.py`

```python
@dataclass
class Rule:
    """Symbolic rule. Generic shape: precondition + action → effect."""
    precondition: Predicate          # function: state → bool
    action_key: int                   # the action this rule applies to
    effect: Effect                    # function: state_before → expected_diff
    confirm_count: int = 0
    refute_count: int = 0
    @property
    def confidence(self) -> float:
        n = self.confirm_count + self.refute_count
        return self.confirm_count / max(n, 1)

class RuleStore:
    rules: list[Rule]
    def suggest(self, state: Frame, action: int) -> float: ...
    def induce(self, state_before, action, state_after) -> Rule | None: ...
    def update_confidences(self, trajectory: list[Transition]) -> None: ...
```

**Initial rule templates** (hand-coded, fill via observation):
- `IF action_K applied at object_position(color=X) THEN object_position changes`
  - Templated by color X and action K.
- `IF action_K applied THEN frame_global changed by ≥N cells`
  - Captures "this action does something."
- `IF action_K is undo (==7) THEN frame restored to previous state`
  - We confirm/refute via direct comparison.
- `IF action_K applied at edge_object(color=X) THEN object disappears`
  - Captures "click on the wall to remove it"-like.

Templates are intentionally simple. They cover the *common* ARC-AGI-3 mechanics (move, toggle, click-to-remove, push, undo). Rare mechanics will fail; that's the bound on what symbolic v3.5 can solve.

### `agents/saliency.py`

```python
class Saliency:
    def score_objects(self, frame: Frame) -> np.ndarray:  # (n_objects,)
    def best_click_target(self, frame: Frame) -> tuple[int, int]: ...
    def score_action(self, state: Frame, action: int) -> float: ...
```

Saliency components:
- **Color rarity**: rarer colors in current frame → more salient.
- **Edge proximity**: objects with `touches_edge=True` (UI buttons) → bonus.
- **Singleton-ness**: small singletons → bonus (often interactable).
- **Distinctive shape**: objects with unusual aspect ratios → bonus.

For `best_click_target`: pick the highest-saliency object's centroid (in display coords).

For `score_action(state, action)`:
- Simple actions (1-5, 7): saliency-uniform (no per-action signal here).
- ACTION6: gets a bonus if there are unexplored salient objects to click.
- Discounts revisited-without-effect actions.

## Honest assumptions and risks

### Assumptions
- **Object-list representation is faithful enough.** We saw on Phase 0b that bt33 buttons are correctly identified. Holds for symbolic grids with distinct colors. May fail for ARC-AGI-3 games using colors at color-rank-3+ as "background-like."
- **Rule templates cover most mechanics.** ARC-AGI-3 games are deterministic and most use simple mechanics (move, click, toggle). Should hold.
- **State hash collapses similar states.** Round-to-2-decimals on centroids means 100×100 unique positions per object. May over-collapse on games where small position differences matter.

### Risks
- **Saliency picks the wrong object.** If the salient object is the *background* (e.g., a large rare-color region), the agent clicks there fruitlessly. Mitigation: require salient objects to be small (size < threshold).
- **State graph blows up.** With many distinct states, the agent never recognizes "I've been here before." Mitigation: aggressive hash collapsing.
- **Rules don't learn fast enough.** ARC-AGI-3 levels with human baseline of 4 actions don't give us 12 probe actions to extract rules. Mitigation: bootstrap with v2.5's undo-probe budget.
- **Skill bank is per-game-id specific.** Skills don't transfer across games (good, that's the point), but the bank also doesn't bootstrap. The agent always starts from scratch.

## Test plan

1. **Tier 1 (3 days):** state graph + rule store (3-4 templates) + saliency. Run on bt11 + bt33 + 5 public games. **Gate: any non-zero level cleared on a game where v2.5 scored 0.**

2. **Tier 2 (2 days):** episode-boundary rule induction (heuristic, no LLM). Cross-level rule transfer. **Gate: rule confidence statistics show generalization across levels of the same game.**

3. **Tier 3 (1 day):** evaluation across all 25 public games. Measure RHAE per game. Compare to v2.5 baseline (all zero) and random baseline (all zero per the earlier 9-game run).

## Decision points

- **End of Tier 1:** if no game improves over v2.5, the rule templates are wrong. Ablate which template helps; iterate.
- **End of Tier 2:** if rule induction doesn't help cross-level, the within-level path is doing the work; ship without rule induction.
- **End of Tier 3:** if v3.5 beats v2.5 on ≥5 games: this is Phase 2 baseline. Move to Kaggle packaging (Phase 5). If <5 games: add novelty predictor (Version C lite, v3.7), retest.

## What does NOT change

- Object-list representation (Phase 0b) is foundational, stays.
- Trajectory format (Phase 0c) stays — useful for any future offline training.
- Phase 1's predictor + checkpoint stay in repo. Demoted, not deleted.

## Out of scope (revisit after v3.5 ships)

- Hungarian matching for the predictor.
- Per-episode local LLM rule summarization.
- Synthetic ARC-AGI-3 corpus generation.
- MCTS with shallow rollouts.
- Algorithm Distillation / cross-game meta-learning.
- Neural novelty predictor (Version C lite).

## Why this is the right architecture (one-page version)

We pivoted because Phase 1 produced an artifact that didn't transfer. The literature (CEL, WALL-E) confirms: ARC-AGI-3-style benchmarks are designed to defeat offline-pretrained agents; the agents that work are online learners. Our 165ms per-action budget rules out the per-step-LLM versions of those approaches. State-graph + rule-store + saliency is the budget-aware online-learning architecture that fits.

The terminal-state predictor we built isn't wasted — it's now an *optional* component (a learned hypothesis generator we may revive in v3.9). The corpus isn't wasted — it's useful training data for *any* offline component we add later. Phase 0b's object-list representation is the single foundational asset that stays load-bearing.

If v3.5 beats v2.5, we have a working agent and a path forward. If it doesn't, the symbolic approach has a known set of failure modes (rule template coverage, saliency pick quality) that are diagnosable. Either way, we'll know within a week.
